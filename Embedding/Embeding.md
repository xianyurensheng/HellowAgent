# 1.向量数据库
## 1.1. 向量索引算法
### 1. 基础概念
- KNN（精确最近领）：对比所有向量，找出最相似的。优点准确，缺点速度慢
- ANN（近似最近领）：通过索引找大概率相似的向量。

**向量数据库都是ANN，主流有：FAISS、HNSW**

### 2. FAISS原理
FaceBook开源的向量检索库，核心思路：**分桶聚类**

1. 把海量向量，分成多个小集群
2. 每个桶有一个中心向量
3. 搜索新向量时，找离它最近的桶
4. 只在桶中找相似向量

**类似于超市，分好了各种区域，在区域中选择想要的东西**

### 3. HNSW原理
比较常用的方法，全称：层次化导航小世界图

1. 先给所有向量搭建多层网络图：高层是大框架，低层是细节
2. 高层确定范围
3. 低层就近匹配

**类似于旅游，没有具体的分类，基于节点连接，并且因为图具有跳远能力，所以速度会比聚类快**

# 2. 主流向量数据库
## 2.1. Milvus（米尔沃斯）
- **定位**：企业级开源分布式向量数据库 
- **优势**：
  - Api友好
  - 性能强
  - 多场景适配
  - 分布式集群
  - 支持多种索引
- **缺点**：本地部署麻烦，吃内存 
- **适应**：企业生产环境、海量向量、正式RAG项目、需要长期维护

## 2.2. Chroma（克柔玛）
- **定位**：轻量级开元向量数据库
- **优势**：
  - 无需部署，内嵌运行，直接pip即可
  - 和LangChain无缝集成，零配置能RAG
  - 自带embeding、文档切分、向量存储一体化
- **缺点**：不适合大量数据，集群弱
- **适应**：个人demo

## 2.3. Redis（向量插件RedisVL）
- **定位**：基于Redis的向量检索
- **不建议**

# 3. Embedding优化
## 3.1. 提升语意准确性
1. 模型选择优化<br>
- 选择合适的模型，中文项目用中文模型<br>
- 不要中英文互相套用<br>

2. 文本预处理<br>
**必须做**
- 1.清洗空白符：去掉多余的换行、空格、制表符
- 2.去掉乱码：PDF提取的特殊符号、乱码
- 3.固定分快大小：200～500字左右
- 4.分快带重叠：chunk_size=500, chunk_overlap=100 （防止一句话被切断）
- 5.统一格式：英文小写、去除多余标点

3. 输入优化
- 问题不要太模糊
- 不要用特殊符号、口语化过重的查询
